## 二、实验理论基础
### 2.1 正向工程与逆向工程
1. **正向工程**：按照需求规格 → 设计 → 编码实现 → 生成新系统，是常规软件开发流程。Android开发即为典型正向工程：编写Java/Kotlin源码 → 编译为Class字节码 → 转换为Dex文件 → 打包资源生成未签名APK → 签名后发布。
2. **逆向工程**：从已有的成品系统出发，反向分析结构、逻辑、实现原理，进而重构设计、修改功能。安卓应用逆向就是对已编译完成的APK进行解析、篡改、重打包。
3. 行业背景：移动广告作弊、恶意篡改、盗版分发等行为大量利用逆向技术，造成巨额经济损失；同时**白帽逆向**也用于安全检测、漏洞挖掘、权限审计等合法场景。

### 2.2 Android应用编译与打包流程
标准Android APK编译打包流程：
1. 源码（`.java`/`.kt`）通过`javac`编译为Java Class字节码；
2. 借助`dx`工具将Class文件转换为Android专属`classes.dex`文件；
3. `aapt`工具编译布局、图片、字符串等资源，生成二进制资源文件与`R`类；
4. 整合Dex、资源、第三方库、`AndroidManifest.xml`，生成**未签名APK**；
5. 使用`jarsigner`工具结合密钥库（keystore）对APK签名，最终得到可安装运行的正式APK。

### 2.3 Android应用逆向流程
标准安卓静态逆向流程：
1. 提取设备中的APK文件；
2. 使用**APKTool**解析APK：还原可读XML资源、`AndroidManifest.xml`、Smali字节码；
3. 可选工具：`dex2jar`将Dex转为Jar包、`JADX`将Jar反译为Java源码；
4. 手动修改Smali代码、资源文件、配置文件；
5. 使用APKTool重新回编译生成新APK；
6. 对新APK执行签名操作；
7. 安装、运行、验证修改效果。

### 2.4 APKTool工具介绍
APKTool是安卓逆向主流工具，核心两大功能：
- **解包（decode）**：将APK解析为`res`资源目录、`smali`字节码目录、配置文件，还原二进制XML为可读格式；
- **重打包（build）**：将修改后的文件重新编译为完整APK。

### 2.5 Smali中间语言基础
Smali是Dex字节码的**可读中间表示语言**，APKTool反编译后所有代码均为`.smali`文件，核心语法要点：
1. **类定义**：`.class`声明类、`.super`指定父类、`.source`关联原始源码文件；
2. **成员变量**：`.field`定义静态/实例成员变量，区分`static`与普通变量；
3. **寄存器操作**：`const`系列指令完成常量赋值；`iget/iput`操作实例成员变量，`sget/sput`操作静态成员变量；
4. **方法调用**：
   - `invoke-static`：调用静态方法；
   - `invoke-virtual`：调用public/protected普通方法；
   - `invoke-super`：调用父类方法；
   - `invoke-direct`：调用私有方法；
5. **日志输出**：可通过调用`Log.d()`等系统方法添加日志，用于代码修改验证。

### 2.6 Android签名机制
Android系统强制要求所有安装的APK必须带有合法签名，作用：
1. 校验APK完整性，防止文件被篡改；
2. 标识应用开发者身份；
3. 同一签名的应用可实现数据共享、版本覆盖安装。
实验中使用两种签名方式：Android默认调试密钥`debug.keystore`、自定义密钥库`keystore`。

## 四、详细实验步骤与操作过程
### 4.1 第一步：APK解包（APKTool decode）
使用APKTool对目标APK进行完整解包，提取资源、配置、Smali代码。
#### 执行命令
```cmd
apktool d app-demo-debug.apk -o nowinandroid_unpack
```
#### 操作说明
1. `d`：decode，代表解包指令；
2. `app-demo-debug.apk`：待解包原始文件；
3. `-o nowinandroid_unpack`：指定解包输出文件夹名称。

#### 解包结果
命令执行无报错，目录下生成`nowinandroid_unpack`文件夹，内部目录结构：
- `AndroidManifest.xml`：应用全局配置、权限、组件声明；
- `res/`：布局、图片、字符串等资源文件；
- `smali/`、`smali_classesX`：应用逻辑代码（Smali字节码）；
- `apktool.yml`：APKTool编译配置文件。

### 4.2 第二步：文件修改（资源 + Smali代码双修改）
结合PPT知识点，完成**资源文件修改**和**Smali代码修改**两项操作。

#### 4.2.1 修改字符串资源（界面文本）
1. 进入路径：`nowinandroid_unpack\res\values\strings.xml`；
2. 打开文件，找到应用首页展示文本，修改原有内容，自定义新文案；
3. 检查XML语法，保证标签闭合、无语法错误，保存文件。

#### 4.2.2 修改Smali代码（添加日志输出）
参考PPT中Smali日志示例，在Activity的点击事件方法中添加`Log`输出：
1. 在`smali`目录中定位页面对应`Activity.smali`文件；
2. 找到按钮点击回调方法（`.method` 声明的函数）；
3. 在方法体内添加Smali日志代码：
   ```smali
   const-string v3, "TestLog"
   const-string v4, "代码修改测试成功"
   invoke-static {v3, v4}, Landroid/util/Log;->d(Ljava/lang/String;Ljava/lang/String;)I
   move-result v3
   ```
4. 保证`.locals`寄存器数量充足，保存Smali文件。

> 原理：该段代码对应Java `Log.d("TestLog","代码修改测试成功")`，运行后可在Logcat中捕获日志。

### 4.3 第三步：重打包（APKTool build）
修改完成后，使用APKTool将目录重新编译为新APK。
#### 执行命令
```cmd
apktool b nowinandroid_unpack -o app-modified.apk
```
#### 操作说明
1. `b`：build，代表重打包指令；
2. `nowinandroid_unpack`：修改后的解包目录；
3. `-o app-modified.apk`：指定输出新APK名称。

#### 打包结果
编译过程无语法报错，在当前目录生成**未签名**的`app-modified.apk`。

### 4.4 第四步：APK签名与签名校验
Android禁止安装未签名APK，本次使用**系统默认调试密钥**完成签名。

#### 4.4.1 执行签名命令
```cmd
jarsigner -verbose -sigalg SHA1withRSA -digestalg SHA1 -keystore C:\Users\mac\.android\debug.keystore app-modified.apk androiddebugkey
```
- 密钥库路径：`C:\Users\mac\.android\debug.keystore`（系统默认调试密钥）；
- 别名：`androiddebugkey`；
- 密钥密码：默认`android`，直接回车或输入密码即可完成签名。

#### 4.4.2 签名完整性校验
执行校验命令，验证APK签名是否有效：
```cmd
jarsigner -verify app-modified.apk
```
输出 `jar verified` 表示签名合法、文件完整。

### 4.5 第五步：安装与功能验证
#### 4.5.1 ADB安装APK
开启安卓模拟器并打开USB调试，执行安装命令：
```cmd
adb install -r app-modified.apk
```
- `-r`：覆盖安装，保留应用数据；
- 终端输出 `Success`，代表安装成功。

#### 4.5.2 功能验证
1. **界面验证**：启动应用，页面展示的自定义字符串正常生效，资源修改成功；
2. **代码逻辑验证**：点击对应按钮，打开Logcat工具，捕获到`TestLog`标签的自定义日志，证明Smali代码修改生效；
3. **稳定性验证**：应用无闪退、无崩溃，原有功能、页面跳转、数据展示全部正常。

## 五、实验结果与分析
### 5.1 实验结果
1. 成功使用APKTool完成目标APK解包，完整还原资源、配置文件与Smali字节码；
2. 顺利修改`strings.xml`资源文件与Smali代码并保存，无语法错误；
3. 重打包生成新APK，编译流程正常；
4. 使用调试密钥完成签名，签名校验通过；
5. ADB安装APK成功，界面文本修改、Smali日志代码均正常生效，应用运行稳定。

**整体结论**：本次实验所有操作步骤全部完成，实验目标达成。

### 5.2 结果分析
1. **APK结构分析**：APK本质是压缩包，经过Android专属编译流程生成，普通解压无法还原可读资源，必须依赖APKTool等专用逆向工具。
2. **Smali语言特点**：Smali语法严谨，寄存器数量、指令类型、方法调用规则出错都会导致重打包失败或应用闪退，和PPT讲解的语法规则完全一致。
3. **签名机制作用**：未签名APK会被系统拦截安装，签名机制是Android重要安全屏障，可有效防范恶意篡改APK传播。
4. **逆向技术两面性**：本次为合法教学逆向，可用于安全检测、学习研究；但不法分子会利用该技术制作盗版、恶意应用、广告作弊程序，带来安全威胁。
5. **加固必要性**：本次实验使用开源调试APK，无加固、代码混淆，逆向难度极低；商业应用必须搭配代码混淆、APK加固、签名校验等防护手段。
